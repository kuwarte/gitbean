# Object Format Reference

## Blob

```
content_to_hash = "blob <byte-length>\0<raw file bytes>"
sha1 = SHA1(content_to_hash)   -> 40 hex chars
stored (zlib compressed) at: .gitbean/objects/<hash[0:2]>/<hash[2:]>
```

### Storage path splitting

First 2 hex chars = folder, remaining 38 = filename.
Reason: avoids dumping 100k+ files in one flat directory.

## Reading an object back out (`cat-file`)

1. Look up `.gitbean/objects/<hash[0:2]>/<hash[2:]>`
2. zlib-inflate the file bytes
3. Find the null byte (`\0`) — everything before it is the header
    - header format: `"<type> <size>"`, e.g. `"blob 12"`
4. Everything after the null byte is the actual content
    - `-p` flag → print the content
    - `-t` flag → print just the type (`header.split(" ")[0]`)
    - `-s` flag → print just the size (`header.split(" ")[1]`)

## Index / staging area (`add`)

**Current approach: simplified plain-text format** (not real Git's binary index — see decision note in `PROGRESS.md`).

Stored at `.gitbean/index`, one line per staged file:

```
<blob-hash> <filename>
<blob-hash> <filename>
...
```

Example:

```
e69de29bb2d1d6434b8b29ae775ad8c2e48c5391 test.txt
a1b2c3d4e5f60718293a4b5c6d7e8f9012345678 test2.txt
```

`add <file>` does three things:

1. Hashes the file's content as a blob (reuses `hash-object`'s logic)
2. Stores it via `ObjectStore.writeObject(...)` if not already stored
3. Adds/updates its `filename -> hash` entry in the index

### Real Git's actual index format (for later, if upgrading)

Binary file, roughly:

```
Header: "DIRC" + version (4 bytes) + entry count (4 bytes)
Per entry: ctime, mtime, dev, ino, mode, uid, gid, file size,
           20-byte SHA1, flags+path-length, filename, null padding to 8-byte boundary
Footer: 20-byte SHA1 checksum of everything above
```

## Tree (not yet implemented)

```
for each entry: "<mode> <filename>\0<20 raw bytes of SHA1>"
```

NOTE: binary, not hex string — this is the tricky part.
